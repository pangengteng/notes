# 认证

## SSO (Single Sign On)

### OAuth2.0
```mermaid
sequenceDiagram
    title 标准 OAuth 2.0 授权码模式（仅授权，无认证）
    participant RO as 资源所有者<br/>(Resource Owner, 用户)
    participant Client as 客户端<br/>(Client App)
    participant AS as 授权服务器<br/>(Authorization Server)
    participant RS as 资源服务器<br/>(Resource Server)

    RO->>Client: 1. 请求访问受保护资源
    Client-->>RO: 2. 重定向到授权端点<br/>Location: https://as.com/authorize?<br/>response_type=code<br/>&client_id=CLIENT_ID<br/>&redirect_uri=REDIRECT_URI<br/>&scope=read_data<br/>&state=RANDOM

    RO->>AS: 3. 在授权服务器认证并授权<br/>(登录 + 同意授权)
    AS-->>RO: 4. 重定向回客户端<br/>Location: REDIRECT_URI?code=AUTH_CODE&state=RANDOM

    RO->>Client: 5. 浏览器访问回调地址（带 code）
    Client->>AS: 6. 后端用 code 换 access_token<br/>POST /token<br/>client_id=CLIENT_ID<br/>&client_secret=SECRET<br/>&grant_type=authorization_code<br/>&code=AUTH_CODE<br/>&redirect_uri=REDIRECT_URI

    AS-->>Client: 7. 返回 access_token<br/>{ "access_token": "ACCESS_TOKEN", "token_type": "Bearer", "expires_in": 3600 }

    Client->>RS: 8. 请求受保护资源<br/>GET /api/data<br/>Authorization: Bearer ACCESS_TOKEN
    RS->>AS: 9. (可选) 验证 token 有效性<br/>(通过 introspection 或签名)
    AS-->>RS: 10. 返回 token 状态（active/inactive）
    RS-->>Client: 11. 返回资源数据

    Client-->>RO: 12. 展示资源
```

### OIDC (Open ID Connect)
```mermaid
sequenceDiagram
    title 标准 OpenID Connect (OIDC) 授权码模式（含身份认证）
    participant User as 用户<br/>(Resource Owner)
    participant Client as 客户端应用<br/>(Relying Party)
    participant OP as 身份提供者<br/>(OpenID Provider, IdP)

    User->>Client: 1. 访问受保护资源或点击“登录”
    Client-->>User: 2. 重定向到授权端点<br/>Location: https://idp.com/auth?<br/>response_type=code<br/>&client_id=CLIENT_ID<br/>&redirect_uri=REDIRECT_URI<br/>&scope=openid profile email<br/>&state=RANDOM<br/>&nonce=RANDOM_NONCE

    User->>OP: 3. 在 IdP 登录并授权<br/>(完成身份认证)
    OP-->>User: 4. 重定向回客户端<br/>Location: REDIRECT_URI?code=AUTH_CODE&state=RANDOM

    User->>Client: 5. 浏览器访问回调地址（带 code）
    Client->>OP: 6. 后端用 code 换令牌<br/>POST /token<br/>client_id=CLIENT_ID<br/>&client_secret=SECRET<br/>&grant_type=authorization_code<br/>&code=AUTH_CODE<br/>&redirect_uri=REDIRECT_URI

    OP-->>Client: 7. 返回 ID Token + Access Token<br/>{<br/>  "id_token": "eyJhbGciOiJSUzI1NiIs...",<br/>  "access_token": "ACCESS_TOKEN",<br/>  "token_type": "Bearer",<br/>  "expires_in": 3600<br/>}

    Client->>Client: 8. 验证 ID Token：<br/>- 签名 (JWS)<br/>- issuer (iss)<br/>- audience (aud)<br/>- nonce<br/>- expiration (exp)

    alt 需要更多用户信息
        Client->>OP: 9. 调用 UserInfo Endpoint<br/>GET /userinfo<br/>Authorization: Bearer ACCESS_TOKEN
        OP-->>Client: 10. 返回用户声明 (Claims)<br/>{ "sub": "123", "name": "Alice", "email": "alice@example.com" }
    end

    Client->>Client: 11. 创建本地会话<br/>(基于 ID Token 中的 sub)

    Client-->>User: 12. 登录成功，返回主页
```

**OAuth2.0 VS. OIDC**
> 💡 一句话总结：
OIDC = OAuth 2.0 + 标准化身份认证层。
所有 OIDC 实现都兼容 OAuth 2.0，但 OAuth 2.0 本身不能安全实现“登录”功能。

| 对比维度 | OAuth 2.0 | OpenID Connect (OIDC) |
|---------|-----------|------------------------|
| **核心目的** | **授权（Authorization）**<br>“你能访问什么资源？” | **身份认证（Authentication）**<br>“你是谁？” |
| **标准规范** | [RFC 6749](https://datatracker.ietf.org/doc/html/rfc6749) | [OpenID Connect Core 1.0](https://openid.net/specs/openid-connect-core-1_0.html)（基于 OAuth 2.0） |
| **是否包含认证** | ❌ 否（仅授权） | ✅ 是（在授权基础上增加认证） |
| **关键令牌** | `access_token`（用于访问资源） | `id_token`（JWT 格式，用于身份认证）<br>`access_token`（可选，用于访问 UserInfo） |
| **用户身份标识** | 无标准字段；依赖平台私有接口（如微信 `openid`） | ✅ 标准化 `sub`（Subject）声明，全局唯一用户 ID |
| **必需 Scope** | 自定义（如 `read:data`） | 必须包含 `openid` |
| **防重放机制** | 无 | ✅ `nonce` 参数（绑定 ID Token 到当前会话） |
| **用户信息获取** | 无标准方式；需调用平台私有 API | ✅ 标准化 `/userinfo` 端点（返回 JSON Claims） |
| **令牌验证** | 资源服务器验证 `access_token`（如 introspection 或 JWT 签名） | 客户端直接验证 `id_token` 签名及声明（无需网络请求） |
| **典型用例** | - 第三方 App 访问 Google Drive<br>- 后台服务调用 API | - “使用 Google 账号登录”<br>- 企业单点登录（SSO） |
| **是否支持 SSO** | ❌ 无法安全实现用户登录 | ✅ 是现代 SSO 的标准协议 |
| **标准化程度** | 授权框架，但身份部分无标准 | 高度标准化，跨平台互操作性强 |
| **代表平台** | 微信、微博、GitHub（仅 OAuth 模式） | Google、Microsoft Entra ID、Auth0、Okta、Keycloak |

**微信扫码登录流程**
> 基于 OAuth 2.0 的授权码模式 实现的 身份认证（Authentication），但它 不是 OpenID Connect（OIDC），而是**微信平台的私有认证扩展。因此，这个时序图只包含 OAuth 2.0 + 微信自定义的身份认证逻辑，并未使用 OIDC。
```mermaid
sequenceDiagram
    participant User as 人类用户
    participant WebFront as Web 前端
    participant WebBackend as Web 后台
    participant AuthPage as 微信授权页面<br/>（浏览器中）
    participant WeChatClient as 微信桌面客户端<br/>（本地进程）
    participant WeChatServer as 微信授权服务器

    User->>WebFront: 1. 点击“微信登录”
    WebFront->>WebBackend: 2. 请求登录 URL
    WebBackend->>WebBackend: 3. 生成随机 state 并存储
    WebBackend-->>WebFront: 4. 返回授权 URL
    WebFront-->>User: 5. 跳转到微信授权页面

    User->>AuthPage: 6. 加载授权页

    alt 场景 A：已安装并登录微信客户端
        AuthPage->>WeChatClient: 7a. 探测本地微信<br/>（通过 weixin:// 或 ActiveX）
        WeChatClient->>WeChatClient: 8a. 验证用户已登录
        WeChatClient-->>AuthPage: 9a. 返回“已登录”信号<br/>（通过本地回调或 JS bridge）

        AuthPage->>User: 10a. 网页内显示“确认授权”<br/>（或自动确认）
        User->>AuthPage: 11a. 点击“允许”（在网页中）

        AuthPage->>WeChatServer: 12a. 通知：用户已确认<br/>（携带 uuid/appid）
        WeChatServer-->>AuthPage: 13a. 返回 code 和 state
    else 场景 B：未登录微信客户端
        AuthPage->>WeChatServer: 7b. 启动轮询：<br/>GET /connect/qrcode/uuid/status
        loop 等待扫码
            WeChatServer-->>AuthPage: 8b. 返回 408
            AuthPage->>WeChatServer: 9b. 继续轮询
        end

        User->>WeChatClient: 10b. 用手机微信扫描二维码<br/>（此处 WeChatClient 指手机微信）
        WeChatClient->>WeChatServer: 11b. 上报授权确认
        WeChatServer-->>AuthPage: 12b. 下次轮询返回 code
    end

    AuthPage-->>User: 13. 重定向到回调地址<br/>?code=AUTH_CODE&state=RANDOM_TOKEN

    User->>WebBackend: 14. 访问回调接口
    WebBackend->>WebBackend: 15. 校验 state
    WebBackend->>WeChatServer: 16. 用 code + appsecret 换 access_token + openid
    WeChatServer-->>WebBackend: 17. 返回 openid
    WebBackend->>WebBackend: 18. 创建本地会话
    WebBackend-->>User: 19. 重定向到主页
```
